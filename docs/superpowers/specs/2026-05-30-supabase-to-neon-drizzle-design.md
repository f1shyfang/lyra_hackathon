# Supabase → Neon + Drizzle Migration — Design

**Date:** 2026-05-30
**Status:** Approved pending user review

## Goal

Port the application's **database layer** from Supabase to **Neon Postgres**, using
**Drizzle** as the ORM. Load the existing Supabase data dump into Neon. Storage
(the `draft-images` Supabase bucket) is **out of scope** — it is not core
functionality and stays on Supabase for now.

Auth is **not used** anywhere in the app, so there is nothing to migrate there.

## Current state

- **DB access:** Supabase JS query builder (`supabase.from('x').select/insert/update/...`)
  via `lib/supabase/{client,server}.ts`.
- **Schema:** 6 tables + 2 enums in the `public` schema:
  - enums: `draft_status` (pending|processing|approved|rejected|shipped),
    `ab_test_status` (draft|running|paused|completed)
  - tables: `drafts`, `ai_personas`, `council_feedback`, `ab_tests`,
    `ab_test_variants`, `ab_test_evaluations`
- **Schema nuances** (the old `types/supabase.ts` was wrong on some):
  - `image_urls` on `drafts` and `ab_test_variants` is **`jsonb`** (default `'[]'`),
    NOT `text[]`.
  - `ab_test_variants.avg_score` is a **GENERATED ALWAYS AS STORED** column
    (`total_score / total_evaluations`) — read-only, never inserted/updated.
  - CHECK constraints: scores 0–100, `epsilon` 0–1, `algorithm` ∈
    {epsilon_greedy, ucb, thompson_sampling, fixed_split}.
  - FKs (all `ON DELETE CASCADE`): `ab_tests→drafts`, `ab_test_variants→ab_tests`,
    `ab_test_evaluations→{ab_tests,ab_test_variants,ai_personas}`,
    `council_feedback→{drafts,ai_personas}`.
  - UNIQUE: `ab_test_evaluations(ab_test_id,variant_id,persona_id)`,
    `ab_test_variants(ab_test_id,name)`,
    `council_feedback(draft_id,persona_id,iteration_number)`.
- **Dump:** plain-text Supabase **cluster** dump at
  `/mnt/c/Users/micha/Downloads/db_cluster-25-12-2025@16-10-19.backup/db_cluster-25-12-2025@16-10-19.backup`.
  Contains Supabase roles, `auth`/`storage`/`realtime`/etc. schemas — only the
  `public` objects are wanted. Data volume: `ai_personas`=9, `drafts`=24,
  `council_feedback`=57; the three `ab_test*` tables are empty.
- **Call sites to migrate (DB only):** ~15 files —
  `app/api/drafts/*`, `app/api/ab-tests/*`, `app/api/simulation/run`,
  `lib/services/{ab-test-engine,council-processor,quality-gate}.ts`,
  and the example/manufacturing pages that query directly
  (`app/examples/{database,ai-personas,simulation}`, `app/test`,
  `app/manufacturing/*`).
- **Keep on Supabase:** `lib/supabase/{client,server}.ts`,
  `lib/storage/image-upload.ts`, `app/examples/storage/page.tsx`, and the
  `@supabase/*` deps — storage only.

## Target architecture

- **Driver:** `drizzle-orm/neon-http` + `@neondatabase/serverless` (HTTP, stateless —
  best fit for Next.js 16 route handlers / serverless).
- **New files:**
  - `lib/db/schema.ts` — Drizzle schema, generated by `drizzle-kit introspect`
    from the loaded Neon DB (guarantees fidelity), then lightly cleaned.
  - `lib/db/index.ts` — exports a typed `db = drizzle(neon(process.env.DATABASE_URL!))`.
  - `drizzle.config.ts` — drizzle-kit config (introspect/generate).
  - `db/neon_import.sql` — cleaned public-only DDL + COPY data + constraints,
    extracted from the dump.
- **Env:** add `DATABASE_URL` (Neon pooled connection string) to `.env.local`
  and `env.example`.
- **Deps:** add `drizzle-orm`, `@neondatabase/serverless`, `drizzle-kit` (dev).
  Keep `@supabase/*` (storage). `dotenv` for any node load script if needed.

## Load strategy

The dump is a full cluster dump, so we do NOT restore it raw (Neon rejects
Supabase roles/extensions). Instead:

1. Install `postgresql-client` (for `psql`) in the dev environment.
2. Build `db/neon_import.sql` containing only `public` objects:
   the 2 `CREATE TYPE` enums, the 6 `CREATE TABLE`s (stripped of `OWNER TO`/
   RLS/grants), the 6 `COPY ... FROM stdin` data blocks (extracted verbatim from
   the dump), and the PK/FK/UNIQUE `ALTER TABLE` constraints.
3. Run it against Neon with `psql "$DATABASE_URL" -f db/neon_import.sql`.
4. Verify row counts (9 / 24 / 57 / 0 / 0 / 0).
5. `drizzle-kit introspect` → generate `lib/db/schema.ts` from the live DB.

## Query rewrite mapping

Supabase query-builder → Drizzle (`drizzle-orm`), e.g.:

| Supabase | Drizzle |
|---|---|
| `supabase.from('drafts').select('*').eq('id', id).single()` | `db.select().from(drafts).where(eq(drafts.id, id)).then(r => r[0])` |
| `.insert({...}).select().single()` | `db.insert(drafts).values({...}).returning().then(r => r[0])` |
| `.update({...}).eq('id', id)` | `db.update(drafts).set({...}).where(eq(drafts.id, id))` |
| `.order('created_at',{ascending:false})` | `.orderBy(desc(drafts.createdAt))` |
| `.range(offset, offset+limit-1)` | `.limit(limit).offset(offset)` |
| `.delete().eq('id', id)` | `db.delete(drafts).where(eq(drafts.id, id))` |

Each route handler/service loses its `createClient()` call and imports `db` +
`schema` + drizzle operators instead. Error handling shifts from Supabase's
`{data, error}` tuples to try/catch around thrown driver errors.

## Out of scope / non-breaking

- Storage code stays untouched and on Supabase.
- The Python sentiment classifier (`api.py`, `prediction_service.py`, etc.) is
  unrelated to Supabase and untouched.
- `types/supabase.ts` may be deleted once `lib/db/schema.ts` is the source of
  truth and no code imports it (the Supabase storage clients don't need it).

## Verification

- `psql` row-count check post-load.
- `npm run build` (Next.js typecheck) passes after the rewrite.
- Spot-check one read and one write path (e.g. GET `/api/drafts`,
  POST `/api/drafts`) against Neon data.

## Risks

- `drizzle-kit introspect` may not perfectly express the generated `avg_score`
  column or all CHECK constraints — review the generated `schema.ts` and patch
  by hand if needed (mark `avg_score` as generated/read-only so inserts omit it).
- `image_urls` is `jsonb` — Drizzle column must be `jsonb`, and code that treated
  it as a JS `string[]` keeps working (jsonb round-trips arrays).
